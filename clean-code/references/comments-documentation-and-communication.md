# Comments Documentation And Communication

## Comment Policy

- Prefer self-explanatory code first.
- Use comments to explain why, constraints, and tradeoffs.
- Remove comments that restate obvious code behavior.

## Documentation Scope

- Document public interfaces and contracts.
- Document assumptions, invariants, and operational limits.
- Keep docs close to code when possible.

## Change Communication

- Write commit and PR descriptions around intent and impact.
- Call out risk, migration steps, and rollback approach.
- Keep architectural decisions traceable.

## Anti-Noise Rules

- Avoid stale TODOs without owner and context.
- Avoid narrative comments that drift from implementation.
- Avoid excessive inline comments used to compensate for poor structure.
